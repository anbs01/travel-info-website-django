# 技术选型调整说明

> 本文档记录旅游门户网站项目从 Python/Django 迁移至 .NET 8 / Razor Pages 的决策过程与技术对比分析。

---

## 一、现状与问题

项目最初由 AI Agent 使用 Python/Django 框架开发，开发过程中暴露出以下核心问题：

**1. 可维护性差**
团队成员不熟悉 Python，无法独立 review AI 生成的代码，也无法自行排查和修复问题，完全依赖 AI，开发节奏被动。

**2. 前端自适应质量不稳定**
AI 生成的 HTML/CSS 模板在响应式布局细节上（断点、间距、图片缩放）经常出现问题，但由于看不懂代码，难以精准定位和纠正。

**3. AI 生成代码质量难以验证**
Python 是动态类型语言，AI 生成的代码缺少类型约束，错误不容易在编写阶段发现，只有运行时才暴露问题。

**4. 细节纠错成本高**
每一处细节问题都需要重新描述给 AI，反复确认，效率低下，工作量远超预期。

---

## 二、为什么换 .NET

| 对比维度 | Python / Django | .NET 8 / Razor Pages |
|---|---|---|
| 团队熟悉度 | ❌ 不熟悉，无法自主维护 | ✅ 团队主力语言 |
| 代码可读性 | ❌ 动态类型，AI生成难验证 | ✅ 强类型，IDE 提示完善 |
| AI 生成质量 | ⚠️ 动态类型出错不易发现 | ✅ 编译期即可发现错误 |
| 自主 Debug 能力 | ❌ 依赖 AI | ✅ 可独立排查 |
| Review 效率 | ❌ 看不懂改不了 | ✅ 能看懂每一行 |

**结论：换 .NET 的核心收益不是技术上更先进，而是团队能掌控代码，能自主 review、纠错、迭代，开发效率才能真正提升。**

---

## 三、自适应问题的本质

> 有观点认为"换框架能解决自适应问题"，这个理解需要纠正。

**自适应（响应式布局）是纯前端问题，与后端语言无关。**

- Django 用 Jinja2 模板渲染 HTML
- Razor Pages 用 Razor 模板渲染 HTML
- 最终输出的 HTML/CSS 结构是一样的

自适应质量取决于：
1. 选用的 CSS 框架（Bootstrap / Tailwind）
2. AI 生成前端代码的质量
3. 团队能否 review 和纠正前端代码

换成 .NET 后，团队能看懂 C# 和 HTML，可以直接介入前端细节的纠正，这才是自适应质量提升的真正原因。

## 四、前端技术方案对比

### 4.1 为什么不用 Vue / React 前后端分离

| 问题 | 说明 |
|---|---|
| SEO 问题 | 客户端渲染默认 SEO 很差，需要额外做 SSR（Nuxt.js），复杂度翻倍 |
| 工作量增加 | 后端变成纯 API，需要额外维护一套 .NET Web API 项目 |
| 维护成本 | 多一套 Node.js / Vue 项目，多一份依赖和部署复杂度 |
| 不适合内容门户 | 旅游门户以内容展示为主，不需要复杂的前端交互 |

### 4.2 CSS 框架选择

**Tailwind CSS（推荐）**
- 目前最主流的 CSS 框架，.NET 官方模板已内置支持
- 自适应断点写法直观（`md:` `lg:`），AI 掌握度极高
- 完全自由定制，不受组件库风格限制，能还原任意设计稿

**Bootstrap 5**
- 老牌稳定，栅格系统成熟，自适应能力强
- AI 熟悉度高，生成代码质量稳定
- 适合快速出页面

两者都不需要 Vue/React，直接在 Razor 模板里使用，简单直接。

## 五、Blazor 方案分析

### 5.1 Blazor Server + 预渲染

有观点认为 Blazor Server 开启预渲染（`ServerPrerendered`）可以解决 SEO 问题，这个说法基本正确，但有细节：

- ✅ 首屏输出完整 HTML，爬虫可以抓取内容
- ⚠️ 页面交互后变为 SPA 行为，内链跳转爬虫不一定能跟进
- ⚠️ 依赖 SignalR 长连接，并发用户多时服务器压力大，有连接数上限
- ⚠️ 对内容门户够用，但不如 Razor Pages 天然干净

### 5.2 Blazor WebAssembly

- ❌ 纯客户端渲染，SEO 很差
- ❌ 首屏加载慢（需下载 .NET Runtime）
- ❌ 完全不适合内容门户网站

### 5.3 Masa Blazor 组件库

Masa Blazor 是基于 Material Design 的 Blazor 组件库，组件丰富，但：

- ❌ Material Design 风格偏管理系统，与旅游门户的视觉气质不符
- ❌ 客户提供了设计稿，组件库风格会与设计稿冲突
- ✅ **适合用于后台管理系统部分**

## 六、最终技术选型

| 层次 | 技术选型 | 理由 |
|---|---|---|
| 后端框架 | .NET 8 / ASP.NET Core | 团队熟悉，强类型，可维护 |
| 前台页面 | Razor Pages | 服务端渲染，SEO 天然友好，无需额外配置 |
| CSS 框架 | Tailwind CSS | 自适应强，AI 生成质量高，完全自由定制 |
| 数据库 ORM | EF Core + MySQL | 与现有数据库兼容，迁移成本低 |
| 后台管理 | Razor Pages（轻量实现） | 需求不复杂，不引入重型框架 |
| 缓存 | OutputCache / Redis | 热门页面缓存，接近静态化效果 |

---

## 七、SEO 友好性说明

客户要求 SEO 友好，Razor Pages 是最直接的解决方案：

- 服务端渲染，每个页面输出完整 HTML，爬虫直接可读
- 无需任何额外配置，天然支持
- 支持自定义 `<title>`、`<meta description>`、Open Graph 标签
- URL 路由清晰，支持 Slug 友好 URL

## 八、迁移策略

Python/Django 版本代码**完整保留**，不删除。

在同一个 Git 仓库下新建 `dotnet/` 目录，存放 .NET 项目：

```
仓库根目录/
├── apps/          ← Python 原有代码，保留
├── config/        ← Python 配置，保留
├── docs/          ← 设计文档，两边共用
└── dotnet/        ← 新建，.NET 项目
    ├── TravelPortal.sln
    └── TravelPortal.Web/
```

**迁移可复用的资产：**
- `docs/` 下的需求文档、设计规范 JSON、设计稿图片
- 数据库模型设计（直接翻译为 EF Core Entity 类）
- 现有 MySQL 数据库（EF Core 支持 MySQL，数据可直接迁移）
